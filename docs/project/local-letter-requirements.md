# 로컬레터 요구사항 명세서

작성일: 2026-06-08

이 문서는 Notion의 `로컬레터 요구사항 명세서` 문서와 동기화한다.
요구사항을 추가, 수정, 완료 처리할 때는 이 파일과 Notion 문서를 함께 갱신한다.

- Notion 문서: https://app.notion.com/p/379d44671e2481ed8b98fd9ba7f38c0f
- 상위 Notion 페이지: SSAFY Enjoy-Trip
- 로컬 문서: `docs/project/local-letter-requirements.md`
- 기준 자료: `docs/project/Local_Letter_Editorial_Map.pdf`, `docs/project/로컬레터 기획안 최종_0606.pdf`, `docs/project/attraction-embedding-target-regions.md`

이 문서는 애자일 프로젝트 백로그로 바로 옮길 수 있도록 사용자 관점의 유저 스토리와 인수 조건 중심으로 정리한다.
우선순위는 `Must`, `Should`, `Could`로 나눈다.

## 구현 상태 표기 규칙

Notion 문서는 유저 스토리와 인수 조건의 원본이며, 로컬 문서는 repo에서 구현 여부를 추적할 수 있도록 상태 열을 추가한다.

| 상태 | 의미 |
|---|---|
| 구현됨 | 현재 backend repo에서 유저 스토리를 직접 만족하는 API, 도메인, 저장소, 테스트 근거가 있다. |
| 부분 구현 | 기존 Enjoy-Trip 기능으로 일부 기반은 있으나 로컬레터 스토리를 그대로 만족하지는 못한다. |
| 미구현 | 현재 backend repo에서 해당 기능을 찾지 못했다. |
| 결정 필요 | MVP 범위, 데이터 정책, 외부 연동 방식 등 제품/기술 결정을 먼저 내려야 한다. |
| 문서화 | 구현 기능이 아니라 요구사항, Ready/Done 기준, 동기화 규칙을 정리하는 항목이다. |

## Epic 1. 오늘 갈 곳 발견하기

| ID | 유저 스토리 | Priority | 상태 | 구현 근거 |
|---|---|---|---|---|
| US-01 | 오늘의 동네 브리핑 보기 | Must | 부분 구현 | `WeatherController`의 `/api/weather/briefings`는 있으나, 현재 날씨와 시기에 맞는 동네 브리핑, `왜 지금 좋은지`, 추천 동네 이유를 묶은 API는 없다. |
| US-02 | 상황 칩으로 추천 좁히기 | Must | 미구현 | `오늘 저녁`, `이번 주말`, `혼자`, `데이트`, `비 오는 날`, `뚜벅이` 같은 상황 칩과 추천 갱신 API를 찾지 못했다. |

### US-01. 오늘의 동네 브리핑 보기

**As a** 목적지를 아직 정하지 않은 사용자
**I want** 현재 날씨와 시기에 맞는 동네 브리핑을 보고 싶다
**So that** 오늘 어디를 가면 좋을지 빠르게 결정할 수 있다.

**Acceptance Criteria**

- Given 사용자가 진행 중인 로컬레터가 없을 때, When 홈에 진입하면, Then 오늘의 동네 브리핑이 표시된다.
- Given 날씨와 시기 데이터가 있을 때, When 브리핑을 생성하면, Then `왜 지금 좋은지`가 포함되어야 한다.
- Given 추천 동네가 표시될 때, Then 단순 장소 목록이 아니라 상황에 맞는 이유가 함께 보여야 한다.

### US-02. 상황 칩으로 추천 좁히기

**As a** 현재 상황이 분명한 사용자
**I want** `오늘 저녁`, `이번 주말`, `혼자`, `데이트`, `비 오는 날`, `뚜벅이` 같은 상황 칩을 선택하고 싶다
**So that** 내 상황에 맞는 동네와 경험만 빠르게 볼 수 있다.

**Acceptance Criteria**

- Given 홈 화면에 진입했을 때, Then 상황 칩 목록이 표시된다.
- When 사용자가 상황 칩을 선택하면, Then 추천 동네와 경험 카드가 선택한 상황에 맞게 갱신된다.
- When 여러 칩을 선택하면, Then 선택 상태가 화면에 명확히 표시된다.

### US-03. 추천 경험 카드 보기

**As a** 오늘 할 일을 찾는 사용자
**I want** `겨울 시장 간식`, `해질녘 한강 산책` 같은 경험 카드를 보고 싶다
**So that** 장소명이 아니라 내가 실제로 할 행동을 기준으로 선택할 수 있다.

**Acceptance Criteria**

- Given 추천 결과가 있을 때, Then 경험 카드는 경험명, 대표 이미지, 지역, 추천 이유를 포함한다.
- Then 경험명은 상호명만으로 구성되지 않고 행동/분위기를 포함해야 한다.
- When 사용자가 경험 카드를 선택하면, Then 경험 상세 화면으로 이동한다.

## Epic 2. 지도에서 로컬 경험 탐색하기

| ID | 유저 스토리 | Priority | 상태 | 구현 근거 |
|---|---|---|---|---|
| US-04 | 지도에서 경험 핀 탐색하기 | Must | 부분 구현 | `/api/attractions`는 지역, 키워드, 좌표, 반경 검색 기반은 제공하지만 경험 중심 핀 라벨 모델은 없다. |
| US-05 | 쪽지 보기 토글 사용하기 | Must | 미구현 | 쪽지 도메인, 쪽지 핀, 지도 토글 API가 없다. |
| US-06 | 관심사별 쪽지 필터링하기 | Must | 미구현 | 쪽지 카테고리와 카테고리별 조회 API가 없다. |
| US-07 | 근처 쪽지 뽑기 | Must | 미구현 | 반경 50m 내 쪽지 랜덤/추천 조회 API가 없다. |

### US-04. 지도에서 경험 핀 탐색하기

**As a** 특정 동네를 둘러보고 싶은 사용자
**I want** 지도에서 경험 핀을 보고 싶다
**So that** 동네 안에 어떤 경험이 있는지 직관적으로 탐색할 수 있다.

**Acceptance Criteria**

- Given 사용자가 지도 탭에 진입하면, Then 지도와 지역 선택 UI가 표시된다.
- When 사용자가 지역을 선택하면, Then 해당 지역의 경험 핀이 지도에 표시된다.
- Then 경험 핀은 장소명보다 경험 중심 라벨을 우선 표시한다.

### US-05. 쪽지 보기 토글 사용하기

**As a** 다른 사람들이 남긴 로컬 이야기를 보고 싶은 사용자
**I want** 지도에서 쪽지 보기 토글을 켜고 끄고 싶다
**So that** 필요할 때만 사람들의 쪽지를 지도 위에서 확인할 수 있다.

**Acceptance Criteria**

- Given 지도 화면이 기본 상태일 때, Then 쪽지 핀은 기본적으로 숨겨져 있다.
- When 사용자가 쪽지 보기 토글을 켜면, Then 지도에 쪽지 핀이 표시된다.
- When 사용자가 토글을 끄면, Then 쪽지 핀이 다시 숨겨진다.

### US-06. 관심사별 쪽지 필터링하기

**As a** 특정 감각의 정보를 찾는 사용자
**I want** 음악, 책, 영화, 꿀팁, 이동 팁 같은 카테고리로 쪽지를 필터링하고 싶다
**So that** 내 관심사에 맞는 로컬 팁만 볼 수 있다.

**Acceptance Criteria**

- Given 쪽지 보기가 켜져 있을 때, Then 쪽지 카테고리 필터가 표시된다.
- When 사용자가 카테고리를 선택하면, Then 해당 카테고리의 쪽지만 표시된다.
- Then 전체 보기로 돌아갈 수 있어야 한다.

### US-07. 근처 쪽지 뽑기

**As a** 지금 내 주변의 이야기를 알고 싶은 사용자
**I want** 현재 위치 근처의 쪽지를 하나 뽑아보고 싶다
**So that** 지도 전체를 탐색하지 않아도 주변의 감각적인 정보를 발견할 수 있다.

**Acceptance Criteria**

- Given 위치 권한이 허용된 상태에서, When 사용자가 근처 쪽지 뽑기를 누르면, Then 반경 50m 내 쪽지 중 하나가 표시된다.
- Given 반경 50m 내 쪽지가 없을 때, Then 쪽지가 없다는 빈 상태와 다른 탐색 액션이 표시된다.
- Then 표시된 쪽지는 저장하거나 쪽지 작성 흐름으로 이어질 수 있다.

## Epic 3. 지역 브리핑과 경험 상세 이해하기

| ID | 유저 스토리 | Priority | 상태 | 구현 근거 |
|---|---|---|---|---|
| US-08 | 지역 브리핑 보기 | Must | 미구현 | 지역 브리핑 aggregate/API가 없다. |
| US-09 | 경험 상세 보기 | Must | 부분 구현 | `Attraction` 상세 기반은 있으나 경험명, 지금 좋은 이유, 추천 이동방식, 관련 쪽지를 묶은 경험 상세는 없다. |
| US-10 | 경험을 로컬레터에 담기 | Must | 부분 구현 | 계획 routeItems 저장은 있으나 경험 카드에서 새/기존 로컬레터 선택으로 이어지는 전용 흐름은 없다. |

### US-08. 지역 브리핑 보기

**As a** 특정 동네가 궁금한 사용자
**I want** 그 동네가 지금 왜 좋은지 브리핑을 보고 싶다
**So that** 평점이나 리뷰만으로 알 수 없는 맥락을 이해할 수 있다.

**Acceptance Criteria**

- Given 사용자가 홈 추천 동네 또는 지도 지역을 선택하면, Then 지역 브리핑 화면으로 이동한다.
- Then 지역 브리핑에는 동네명, 현재 날씨, 오늘의 결론, 지금 좋은 이유가 포함된다.
- Then 이동방식별 팁과 추천 경험 카드가 함께 표시된다.

### US-09. 경험 상세 보기

**As a** 마음에 드는 경험을 발견한 사용자
**I want** 경험 상세에서 지금 좋은 이유와 관련 정보를 보고 싶다
**So that** 이 경험을 내 일정에 담을지 판단할 수 있다.

**Acceptance Criteria**

- Given 사용자가 경험 카드를 선택하면, Then 경험 상세 화면이 열린다.
- Then 경험 상세에는 경험명, 실제 장소명, 대표 이미지, 지금 좋은 이유, 현재 날씨, 추천 이동방식이 표시된다.
- Then 함께 담기 좋은 경험과 관련 쪽지를 확인할 수 있다.

### US-10. 경험을 로컬레터에 담기

**As a** 경험을 저장하고 싶은 사용자
**I want** 경험 카드를 내 로컬레터에 담고 싶다
**So that** 나중에 여행 계획으로 이어갈 수 있다.

**Acceptance Criteria**

- Given 사용자가 경험 상세를 보고 있을 때, Then `내 로컬레터에 담기` 액션이 표시된다.
- When 사용자가 담기를 누르면, Then 새 로컬레터 생성 또는 기존 로컬레터 선택 흐름이 제공된다.
- Then 저장 완료 후 사용자는 저장된 위치를 확인할 수 있다.

### US-11. 조건을 선택해 로컬레터 만들기

**As a** 직접 여행 계획을 만들고 싶은 사용자
**I want** 언제, 누구와, 어떤 분위기, 어떤 이동 방식으로 갈지 선택하고 싶다
**So that** 내 상황에 맞는 맞춤 로컬레터를 추천받을 수 있다.

**Acceptance Criteria**

- Given 사용자가 새 로컬레터 만들기를 시작하면, Then 시기/기간, 동행자, 분위기, 이동방식 입력 단계가 제공된다.
- Then 지역은 직접 선택하거나 `아직 모르겠어요`로 추천받을 수 있어야 한다.
- When 필수 조건을 입력하면, Then 추천 결과 화면으로 이동한다.

### US-12. 추천 로컬레터 미리보기

**As a** 추천 결과를 확인하고 싶은 사용자
**I want** 추천 동네와 추천 흐름을 미리 보고 싶다
**So that** 로컬레터를 만들기 전에 내 취향에 맞는지 판단할 수 있다.

**Acceptance Criteria**

- Given 사용자가 조건 입력을 완료하면, Then 추천 동네 목록과 추천 흐름이 표시된다.
- Then 추천 흐름은 경험의 순서와 간단한 이유를 포함한다.
- When 사용자가 생성 버튼을 누르면, Then 로컬레터가 저장되고 상세 화면으로 이동한다.

### US-13. 로컬레터 목록 관리하기

**As a** 여러 여행 계획을 가진 사용자
**I want** 예정된 레터, 진행 중인 레터, 다녀온 레터를 구분해 보고 싶다
**So that** 내 여행 계획과 기록을 쉽게 관리할 수 있다.

**Acceptance Criteria**

- Given 사용자가 내 레터 탭에 진입하면, Then 예정된 레터, 진행 중인 레터, 다녀온 레터가 구분되어 표시된다.
- Given 아직 만든 로컬레터가 없을 때, Then 빈 상태와 새 로컬레터 만들기 CTA가 표시된다.
- When 사용자가 로컬레터를 선택하면, Then 로컬레터 상세 화면으로 이동한다.

### US-14. 로컬레터 편집하기

**As a** 추천받은 코스를 조정하고 싶은 사용자
**I want** 로컬레터의 경험을 추가, 삭제, 순서 변경하고 싶다
**So that** 추천 코스를 내 실제 일정에 맞게 고칠 수 있다.

**Acceptance Criteria**

- Given 로컬레터 상세에서, When 사용자가 수정하기를 누르면, Then 편집 모드로 진입한다.
- Then 사용자는 경험을 추가하거나 삭제할 수 있다.
- Then MVP에서는 위/아래 이동 버튼 또는 간단한 순서 변경 방식으로 경험 순서를 바꿀 수 있다.
- When 저장하면, Then 변경된 로컬레터 상세가 표시된다.

## Epic 5. 여행 중 로컬레터 경험하기

- Given `backend/batch/src/main/resources/embedding-target-regions.yml`이 있을 때, Then 지정된 지역 외 entry가 있으면 live batch는 실행되지 않아야 한다.
- Then 현재 문서 기준 허용 지역은 강릉시(`sido_code=32`, `gugun_code=1`)와 전주시(`sido_code=37`, `gugun_code=12`)이다.
- Then 실제 MVP 지역을 서울 동 단위로 갈지 강릉/전주 데이터 축으로 갈지는 별도 결정 사항으로 남긴다.

## Definition of Ready

- 유저 스토리에 사용자, 목표, 가치가 명확히 적혀 있다.
- 인수 조건이 Given/When/Then 또는 검증 가능한 문장으로 작성되어 있다.
- 필요한 데이터 출처와 API 의존성이 식별되어 있다.
- 화면 또는 API가 필요한 스토리는 관련 진입 경로가 적혀 있다.
- Must 스토리는 MVP 데모 시나리오와 연결되어 있다.

## Definition of Done

- 스토리의 모든 인수 조건을 만족한다.
- 백엔드 repo에서 구현하는 스토리는 화면 검증 대신 실제 실행 중인 백엔드에 HTTP 요청을 보내 응답 JSON을 확인한다.
- API 응답은 `success`, `data`, `error` 공통 envelope와 핵심 data field가 기대한 값과 shape로 내려오는지 검증한다.
- 생성, 수정, 삭제 기능은 요청 후 조회 API를 다시 호출해 실제 데이터 변경이 응답 JSON에 반영되는지 확인한다.
- 빈 상태, 로딩 상태, 실패 상태가 처리되어 있다.
- 백엔드 API 또는 데이터 연동이 필요한 경우 기본 테스트가 통과한다.
- 데모 플로우에서 사용자가 다음 행동을 막힘 없이 이어갈 수 있다.

## 결정 필요 사항

- 제품 기획의 초기 MVP 지역은 `망원·합정`, `성수·서울숲`, `을지로·충무로`인데, repo의 embedding target 문서는 `강릉시`, `전주시`를 첫 대상 지역으로 둔다. 실제 MVP 데이터 축을 확정해야 한다.
- 친구/같이 여행하기는 기획서에서 고민 항목으로 남아 있어 MVP 포함 여부를 결정해야 한다.
- 외부 SNS/블로그/유튜브/X/당근 등에서 로컬 정보를 수집하는 방식은 기술 가능성, 정책, 저작권 검토가 필요하다.
- AI 추천과 브리핑을 운영자가 어디까지 검수할지 결정해야 한다.
- 위치 기반 쪽지의 공개 범위, 신고 정책, 보관 기간을 정해야 한다.
